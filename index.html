Purpose of this document

This document provide the audience with OpenAPI specification for describing HSBC QR code collection's REST APIs. The scope of this document is limited to digital payments in Singapore.

The target audience of this document is the Developer, Business Analyst and other related Project Team Member (who has the basic technical know-how of Web technology such as REST or JSON) of HSBC’s client (i.e. the Merchant)

Getting Started

Step 1: Contact HSBC

In order to setup and start using PayNow specific APIs, please contact the HSBC team to acquire all prerequisites which are covered in Step 2.

Step 2: Get All Prerequisites

Client ID & Client Secret

What is the purpose?

  • These credentials are for API Authentication. Format is defined in Authentication section.

Where to get it?

  • HSBC will deliver the Client ID and Client Secret to the Merchant via secure email during the Merchant onboarding procedure.

How to use it?

  • The Merchant is required to put the Client ID and Client Secret into HTTP header every time when calling the API.

Merchant Profile & Merchant ID

What is the purpose?

  • The Merchant profile stores all the required Merchant information in the HSBC backend system in order to enable the payment service. The Merchant ID is for Merchant identification in each API call.

Where to get it?

  • During the onboarding procedure, HSBC will gather the Merchant information and set up the Merchant profile. Once the profile is setup HSBC will provide the Merchant ID to the Merchant via secure email.

How to use it?

  • The Merchant is required to put the Merchant ID into the field merId for Merchant identification every time when calling the Payment Request API.

DNS Setup

What is the purpose?

  • The HSBC network security needs to whitelist the Domain Name of the Merchant system in order to enable the Status Notification API (in which the request is made from HSBC to Merchant).

Where to get it?

  • The Merchant has to provide DNS information to HSBC during onboarding procedure. This information can be provide by the Merchants IT team.

Find below the flow diagram of the Merchant Profile & DNS Setup for your reference:

X.509 Public Key Certificate

What is the purpose?

  • HSBC and the Merchant need to exchange Public Key Certificate for Message Encryption purpose.

Where to get it?

  • The Merchant is required to procure a public certificate from an authorized CA in Production environment.
  • Public Certificate Specification:
    • Certificate type: X.509 version 3
    • Signing algorithm: 2048-bit RSA and
    • Hashing algorithm: SHA-256.
  • For security purposes, the Merchant is required to renew the Public Certificate every one year. For details, please read Key Renewal section.
  • Find below the key exchange process flow for reference: 
    <p><img src="images/key_setup.png"></p>

How to use it?

  • The Merchant is required to use HSBC's Public Key to encrypt a message every time they send request to HSBC, verify a signed message every time they received response from HSBC and vice versa.

    The details will be covered in Message Encryption section.

keyID of JWS & JWE

What is the purpose?

  • HSBC uses JSON Web Signature (JWS) standard to sign a message together with JSON Web Encryption (JWE) standard to encrypt the signed message. Find below the usage of keyID during different processes for reference:

    Process / Object | KeyID of JWS | KeyID of JWE ---|---|--- Signing Message | Used for Create JWS Object Header | n/a Message Encryption | n/a | Used for Create JWE Object Header Message Decryption | Will be extracted as an key alias to bind the corresponding private key | n/a Signature Verification | n/a | Will be extracted as an key alias to bind the corresponding public key

Where to get it?

  • The value of the KeyID is expected to be pre-agreed between HSBC & the Merchant.

How to use it?

  • The conceptual usage has been covered in the previous section. To see the detail of how to invoke keyID during program coding, please fint it in Message Encryption section.
  • Furthermore, during Key Renewal, KeyID should be updated and make sure it can binds to the new corresponding private/public key, for details, please see Key Renewal section.

SSL Connection

  • The Merchant will have to setup HTTPS connection (TLS 1.2) to allow secured API messaging.

Step 3: Test in Sandbox

The Merchant can test and simulate API calls in the HSBC sandbox. Find below the sample API testing flow:

<p><img src="images/api_flow.png"></p>

  2. The Merchant will submit QR Code Creation API request.
  3. HSBC back office system will generate QR Code token.
  4. Return the QR Code token via QR Code Creation API response.
  5. The Merchant will trigger Payment Simulation API in order to simulate a payment submission.

  9. The Merchant can submit a Status Enquiry API to check payment status any time after a payment request is submitted or didn’t found acknowledge message returned after a certain period of time.
  10. Return payment status via Status Enquiry API response.

Message Encryption

Javascript Object Signing and Encryption (JOSE), is a framework intended to provide a method to securely transfer information between parties. The JOSE framework provides a collection of specifications, including JSON Web Signature (JWS) and JSON Web Encryption (JWE), to serve this purpose. HSBC uses JWS to sign message and JWE to encrypt the signed message.

Sign and Encrypt Message before Submit Request

To help you understand how to construct a Signed and Encrypted Message, let's take the Java program below as an example. Do not worry if you are not familiar with Java, the idea is to let you know the steps and all needed components:

private JWSObject signMessage(String messagePayload, KeyStore ks, String keyAlias, String keyPw)
              throws UnrecoverableKeyException, KeyStoreException, NoSuchAlgorithmException, JOSEException {
            #1  Payload payload = new Payload(messagePayload);
            
            #2  JWSHeader header = new JWSHeader.Builder(JWSAlgorithm.RS256).keyID("0001").build();
            #3  JWSObject jwsObject = new JWSObject(header, payload);
            
            #4  PrivateKey privateKey = (PrivateKey) ks.getKey(keyAlias, keyPw.toCharArray());
                JWSSigner signer = new RSASSASigner(privateKey);
            #5  jwsObject.sign(signer);
            
                return jwsObject;
            }
  1. Prepare your Message Payload, that is, the plain json request message
  2. Create JWS Header using RS256 signing algorithm and JWS keyID, in this case, 0001
  3. Create JWS Object by combining JWS Header and Message Payload
  4. Retrieve your Private Key as the signer
  5. Create Signed JWS Object by signing it with the Private Key

Next, you are going to Encrypt the Signed JWS Object:

public JWEObject getEncryptedJWEObject(JWSObject jwsObject, RSAPublicKey key)
              throws JOSEException {
            #1  final Payload jwepayload = new Payload(jwsObject.serialize());
            
            #2  final JWEHeader jweheader =
                  new JWEHeader.Builder(JWEAlgorithm.RSA_OAEP_256, EncryptionMethod.A128GCM).keyID("0001").build();
            
            #3  final JWEObject jweObject = new JWEObject(jweheader, jwepayload);
            
            #4  final JWEEncrypter encrypter = new RSAEncrypter(key);
            #5  jweObject.encrypt(encrypter);
            
                return jweObject;
            }
  1. Prepare your JWE Payload, that is, the Signed JWS Object
  2. Create JWE Header. The algorithm used to encrypt the message body is A128GCM while the algorithm used to encrypt the encryption key is RSA_OAEP_256. JWE keyID is 0001 again.
  3. Create JWE Object by combining JWE Header and JWE Payload
  4. Retrieve HSBC's Public Key as the encrypter
  5. Create Encrypted JWE Object by encrypted it with HSBC's Public Key

Yes, you are now ready to put the Encrypted JWE Object as the message body (you may need to first serialize it into String format, depends on your program code design) of any API call.

Decrypt Message and Verify Signature after Receive Response

Let's look into the following example to see how you decrypt a response message from HSBC:

private String decryptMessage(String respMsgPayload, KeyStoreFactory keyStore)
              throws KeyStoreException, NoSuchAlgorithmException, CertificateException, IOException, 
              java.text.ParseException, UnrecoverableKeyException, JOSEException {
            #1  JWEObject jweObject = JWEObject.parse(respMsgPayload);
            
            #2  PrivateKey privateKey = (PrivateKey) keyStore.getPrivateKey("merchant_private_key_alias");
            
                JWEDecrypter decrypter = new RSADecrypter(privateKey);
            #3  jweObject.decrypt(decrypter);
            
            #4  String signedMessage = jweObject.getPayload().toString();
                return signedMessage;
            }
  1. Create Encrypted JWE Object by parsing the encrypted response message payload
  2. Retrieve Private Key as the decrypter
  3. Decrypt the JWE Object using your Private Key
  4. Get the Signed Message from the decrypted JWE Object

You are now able to extract the plain json message. Yet, before that, you must verify the signature to guarantee data integrity.

private String verifySignature(String signedMessage, KeyStore ks, String keyAlias)
              throws KeyStoreException, JOSEException, ParseException {
            #1  JWSObject jwsObject = JWSObject.parse(signedMessage);
            
                Certificate certificate = ks.getCertificate(keyAlias);
            #2  JWSVerifier verifier = new RSASSAVerifier((RSAPublicKey) certificate.getPublicKey());
            
            #3  if (!jwsObject.verify(verifier)) {
                  throw new ValidationException("Invalid Signature");
                }
            #4  return jwsObject.getPayload().toString();
            }
  1. Create JWS Object by parsing the Signed Message
  2. Retrieve HSBC’s Public Key as the verifier
  3. Verify the signed JWS Object. Invoke error handling if invalid signature found (depends on your code design)
  4. Get the plain json message for further actions

Key Renewal

For security purposes, Merchant is required to renew Public Certificate every one years. Please see the renewal flow for reference:

<p><img src="images/key_renewal.png"></p>
  1. HSBC to send email alert to the Merchant before the original Public Certificate (in this example, with keyID 0001) is about to expire, usually 90 days prior to expiry
  2. The Merchant to renew or purchase new Certificate with, recommended 2 years validity
  3. The Merchant to contact HSBC to arrange Certificate exchange
  4. HSBC to import new the Merchant's Public Certificate. To facilitate a seamless key renewal, the original and new the Certificate can coexist in HSBC system. HSBC system can detemine which Certificate to be used by extracting the keyID within the Merchant's request message.
  5. HSBC to notify the Merchant when the new Certificate is ready
  6. The Merchant to make modifications in their App to point to new Key e.g.with new keyID 0002 during JWS/JWE creation
  7. The Merchant to send encrypted API request with new keyID 0002
  8. HSBC to extract keyID from JWS/JWE then pair with the corresponding key during signature verification

Data Type Overview

Data Type Control:

Data Type Allowed Characters Definition & Important Notice
String (For general field) AlphaNumeric and Symbols General field means field which is NOT a critical field. HSBC will validate all characters received in order to assess any security vulnerability such as, Cross-site Scripting. We will highly recommend the use of just Alpha Numeric characters.
String (For critical field) 0-9 a-z A-Z - _ . Critical field is used to be either a key or search criteria in replace_me backend system and hence tight restriction is applied to the allowed characters. Moreover, the starting and ending space of the string value will be trimmed before stored in HSBC system. For example, string " example 12 34 " will be trimmed to "example 12 34". List of Critical Fields: txnRef merId posMachineId employeeId
Integer 0-9 Instead of having Max Length check for String, integer range will be checked, e.g. 0 = x = 9999

Field Mandatory Control:

Field Mandatory Type Definition & Important Notice
Mandatory Annotated with Required in field definition section. Field & value must be present in the request with valid JSON format.
Optional Without a Required tag in field definition. If you do not wish to pass optional fields, your handler should not pass empty strings {"example":""} nor blank value {"example":" "}.
Conditional Required under a specific condition whose logic is always provided in the field definition if it is a Conditional Field.

Time Zone Control:

Aspect Format Definition & Important Notice
In Request Message yyyy-MM-dd'T'HH:mm:ssZ Time zone is expected to be GMT+8 (Singapore local time). Merchant is required to perform any necessary time zone conversion before submit request if needed.
In Response Message yyyy-MM-dd'T'HH:mm:ss±hh:mm Timezone returned in api_gw object is generated from replace_me API Gateway which located in Cloud and hence is calculated in GMT+0. On the other hand, time field in response object will be returned together with timezone information. For more details, please read each field definition carefully.

FAQ

SSL Connection Questions

Where can I find HSBC SSL server certificates?

The Merchant’s developer is able to export SSL server certificates that has been installed in the browser. To download the certificate, visit the domain of the corresponding API endpoint in using internet browser.

For example, to get the SSL certificate of sandbox environment, use domain name https://devcluster.api.p2g.netd2.hsbc.com.hk/

Message Encryption Questions

What certificates will I need to work for Message Encryption in HSBC's sandbox and production environments?

A self-sign certificate is accepted in sandbox environment. In production, Merchant is required to procure a public certificate from an authorized CA.

Callback Payment Notification API

POST /<Callback_URL>

Payment status will be returned to the Merchant by asynchronous callback once HSBC receives a payment request. Once HSBC receives the payment result from the clearing system, it will push the result back to the Merchant by calling this API.

Implementation HSBC will trigger this API call and defines the interface with OpenAPI standard. The Merchant is required to provide implementation.

Retry Mechanism If unsuccessful response, 4 retries will be triggered in every 2 minutes. There will be maximum 5 API calls including the 1st attempt.

Endpoint Definition Field notifyUrl from Payment Request API will be used as URL endpoint of the corresponding transaction.

Exception Handling Only successful case will be returned. The Merchant can submit an Enquiry API request if there is no acknowledge message returned after a certain period of time.

900020 Merchant ID Not Found
900030 Report Date Invalid (Invalid Format or Later than today)
  • Other than "000000", all other return codes indicate a fail case.
proMsg: string (up to 128 chars)

Corresponding Text Message of Process Return Code

Example
{
  "proCode": "000000",
  "proMsg": "Report Request Successful"
}
