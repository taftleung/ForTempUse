Purpose of this document

The purpose of this document is to describe the technical integration between merchants and HSBC platform to faciltitate the collection of digital payments (i.e. Mobile Collection Service) by means of Open API technology. It depicts the process flow and detailed API functionalities which allows easy and quick integration for merchants.

Assumptions and Prerequisites

The merchant should have the following pre-requites in place before integrating with HSBC platform.

  1. Merchant has the capability to host and consume APIs. HSBC will provide the integration kit / library (for Java, C# & PHP) to facilitate the technical integration work and API triggering.
  2. Merchant is onboarded properly on HSBC platform (via Aggregator) wherein Merchant IDs and a client ID & Client secret key pair will be generated by HSBC and assigned to the merchant.
  3. Merhant to provide all Merchant IDs for HSBC to maintain on platform (Message validation will be based on the relationship of Client ID, Merchant ID)
  4. Merchant needs to acquire public certificate and exchange it with HSBC as part of the integration setup, to support authentication checking in every API call with HSBC platform.
  5. Merchant connects with HSBC over HTTPS (SSL) to allow secured API messaging.

Solution Overview

The following services are available for Merchant to consume:

  1. PC Payment
  2. Mobile Payment
  3. Payment status enquiry
  4. Payment cancellation
  5. Payment status notification
  6. Direct debit- binding/un-binding
  7. Binding status notification

PC Payment

This API is suitable for buyer to consume in Merchant online shopping mall, and conduct Alipay/Wechat QR code payment or debit/credit cards for online banking/quick payment.

For QR code payment, HSBC payment platform returns the QR Code information to Merchant once received it from Alipay/Wechat. The merchant displays the QR Code to Buyer. Buyer scans and enters the information to complete the payment.

For Cards payment, HSBC payment platform returns the cashier's page information to the merchant and the merchant displays. Buyer enters information at the cashier's page to complete the payment.

Mobile Payment

This API is suitable for buyer to use a mobile phone to consume at a merchant APP store, and use Alipay/WeChat pay or quickly pay.

For Wechat APP, Alipay APP and quick payment, HSBC payment platform returns the cashier's page information to merchant, merchant displays the casher's page, buyer enters information on the cashier page to complete the payment.

For WeChat public account, WeChat applet and Alipay service window, HSBC payment platform returns pre-payment information to merchant, merchant evokes WeChat/Alipay, and buyer inputs information to complete the payment.

Payment status enquiry

If the merchant does not receive HSBC payment status notification within a certain period, merchant can initiate the payment status enquiry request to HSBC payment platform. HSBC Payment platform matches the original payment based on the original order number, and forwards the request to Fuiou to enquiry payment status, after receives the response, HSBC Payment Platform returns the result to the merchant.

Business Flow is here: IMAGE

Payment cancellation/Refund

When a refund occurs due to buyer or merchant within a period after the transaction occurs, merchant can refund the payment to the buyer through the refund interface. Once HSBC payment platform received and verified the refund request, payment service provider will refund the original payment to the buyer's account. The total refund amount cannot exceed the total amount of the original order.

Business Flow is here: IMAGE

Payment status notification

The payment status is notified to merchant by asynchronous callbacks. After HSBC payment platform receives the payment result from aggregator, HSBC payment platform will push the payment status to merchant. If merchant does not receive the payment status notification, the payment status enquiry request can be initiated.

After merchant receive the payment status notification and returns SUCCESS to HSBC. If HSBC has not received the SUCCESS returned by merchant, the payment status notification will be periodically pushed to Merchant.

Notice: Up to 5 times, every 2 minutes, and merchant need to provide the URLs to receive the notification

Business Flow is here: IMAGE

Direct debit - binding/un-binding

This API is suitable for buyer to bind/unbind their bank card to one particular Merchant, so the same Merchant can direct debit that bank card for service provided to buyer.

Binding result will be sent to Merchant via Binding status notification API.

On the other hand, the un-binding status will be returned to merchant via the response message of Direct debit Binding/un-binding API.

Business Flow is here: IMAGE

Binding status notification

The binding status will be pushed to merchant system by asynchronous callbacks via Binding status notification API. After HSBC payment platform receives the binding result from aggregator, HSBC payment platform will push the binding status with protocol Number to merchant.

After merchant receive the binding status notification, merchant must return SUCCESS response back to HSBC. If HSBC does not receive the SUCCESS returned by merchant, the payment status notification will be continuously (up to 5 times, every 2 minutes ) pushed to Merchant. After max 5 times, HSBC will stop sending notification status to merchants, if there is still no SUCCESS response send back to HSBC.

Notice: Merchant need to provide the URLs to receive the notification

Integration AT-A-Glance

Here is a quick walkthrought of tasks that should be done before consuming HSBC API.

Client ID & Client Secret

a. Merchant Raise request to HSBC API team for Client ID/Secret b. HSBC API team Create Client ID/Secret for a specific merchant c. HSBC API team Share the Client ID/Secret to Merchant via secure email (to merchant contact point provided by business) d. Merchant Store the Client ID/Secret in application and put it in HTTP header when consume HSBC API

Maintain Merchant's Cert (Steps to import customer key)

a. Merchant Generate Merchant public cert. Please follow the instruction under Cert generation section in Appendix b. HSBC Import Merchant public cert into HSBC Public Keystore (with Key ID, i.e. version number, e.g. 0001 – start from 0001, when renew cert, 0002, 0003….)

# Import Public cert
keytool -import -alias <ClientID_KeyID> -keystore HSBC_<env>_pub.jks -file merchant1.cer

Onboard to HSBC Mobile Collection service

a. Merchant Engage HSBC to onboard HSBC Mobile Collection service. Please engage HSBC customer service team for the detail steps. b. Aggregator Assign Merchant IDs (including all sub Merchant IDs if have) and share to merchant c. Merchant Store this Merchant ID in system and refer it according to the instruction in this document, fill in the Merchant Onboard form and share to HSBC d. HSBC Assign a Client ID to merchant, maintant the relationship for Merchant IDs with Client ID in HSCB API Platform. e. Merchant Send merchant URL(for payment result notification) to HSBC for system configuration. If merchant requires PC payment(Cards payment) and mobile payment function, merchant URL(which triggers cashier's page provided by aggregator) is also required to configure in HSBC system.

Data Type Overview

Data Type Control:

Data Type Allowed Characters Remarks
String 0-9 a-z A-Z and All Symbols HSBC system will execute characters checking upon all string fields we received in order to tackle security vulnerability, such as Cross-site Scripting. Yet, we recommend you to try use AlphaNumeric only for most cases.
Integer 0-9 OpenAPI standard supports Integer field, however, there is NO integer field used in China Mobile Collection at this moment.

Field Mandatory Control:

Field Mandatory Type Definition & Important Notice
Mandatory Annotated with Required in field definition section. Field & value must be present in the request with valid JSON format.
Optional Without a Required tag in field definition. If you don't want to pass fields that are optional, your handler should not pass neither empty strings {"example":""} nor blank value {"example":" "}.
Conditional Required under a specific condition whose logic is always provided in the field definition if it is a Conditional Field.

Time Zone Control:

Aspect Format Definition & Important Notice
In Request Message yyyyMMddHHmmss Time zone is expected to be GMT+8 (China local time). Merchant is required to perform any necessary time zone conversion before submit request if needed.
In Response Message yyyy-MM-dd'T'HH:mm:ss±hh:mm Timezone returned in common object is generated from HSBC API Gateway which located in Cloud and hence is computed in GMT+0.
