How to Read this Document

This document will use a Typical Use Case to walk through the API usage concept and list the key idea by section. There is also a FAQ and List of Schema used by the API functions.

Typical Use Case

Throughout this document, we will use the below Typical Use Case where a Buyer visits a Merchant’s Online Store to Place Order and Make Payment with or without Scanning the Merchant-Presented QR Code which is based on HKMA and HKICL standard (also known as Common QR code specification)*.

For the case where the Buyer needs to scan the QR code shown on Online Web Store, it is commonly known as e-Commerce use case

There are other use cases that may not involve QR Code Image generation such as this same use case conducted in Mobile Applications.

In whatever use case, the required API functions are all documented in this document.

Note: HKICAL's url or website for Common QR Code Specification for Retail Payments in Hong Kong (Merchant-Presented mode) is here.

HKMA's url or website for Common QR Code Specification for Retail Payments in Hong Kong is here.

Please note those urls or websites do not belong to replace_me, use it on your own discretion. By clicking those urls or websites, it means you accept this terms and conditions.

According to this use case, we will explain the 2 vital API elements:

  1. Connectivity – how the parties in the business context of the typical use case connect with each other:

    • Authentication using Client ID
    • Transport Layer Security via SSL Certificate
    • Data or Message Security with Encryption/Decryption based on X.509 Certificate (SSL TLS1.2)

  2. Operations – the actions performed via the API functions to achieve the purpose under the business concept:

    • Outbound API
    • Inbound API

  10. Return payment status via Status Enquiry API response

Connectivity AT-A-Glance

Connectivity refers to establishing the connection between 2 parties (in this context, it refers to Merchant and replace_me) so that the 2 parties can communicate via the API functions and procedures.

Connectivity can be further viewed as Authentication and Security.

Authentication deals with knowing the authenticity of both parties (commonly known as sender and receiver depending who initiate the communication).

Encryption/Decryption deals with data security by making in confidential to only the sender and receiver.

Authentication

Client ID & Client Secret

Client ID is the unique identifier of you when establishing connection to replace_me for making API request Client Secret server similar function as password in layman understanding.

  • replace_me will deliver the Client ID and Client Secret (both in text/plain format) to Merchant via secure email during Merchant onboarding process.
  • Merchant is required to put the said pair into HTTP header every time when consumes API for API Authentication. Format is defined as below.

Merchant Profile (Merchant ID)

Merchant ID is another unique identifier from Business Context after you successfully connected or authenticated to replace_me by your Client ID.

  • replace_me will deliver the Merchant ID to Merchant via secure email during Merchant onboarding prcoess.
  • Merchant is required to put the Merchant ID into field merId every time when submits Payment Request API for Merchant identification.
  • Detail of such use case will be explained later in Outbound API section.

Transport Layer Security – SSL Certificate

The connection between merchant and replace_me counts on the security protocol to encrypt communication - SSL (Secure Sockets Layer) via HTTPS over the Internet. SSL protocols use what is known as an asymmetric Public Key Infrastructure (PKI) system. SSL Digital Certificate is used when merchant requests a HTTPS connection to replace_me. This certificate contains the public key needed to begin SSL handshake which involves the generation of shared secrets to establish a uniquely secure connection between merchant and replace_me. In summary, with a HTTPS connection, all communications are securely encrypted between you as merchant and replace_me. This is commonly known as Transport Layer Security or (communication) channel security, where a “tunnel” is establish between two parties (merchant and replace_me) so as to shield off the peeping eyes.

Data or Message Security - Encryption/Decryption

On top of the Transport Layer Security, replace_me adopts additional security on the message being passed through the connection session – namely Message or Data Security. Javascript Object Signing and Encryption (JOSE), is a framework intended for this said purpose to provide a method to securely transfer information between parties. The JOSE framework provides a collection of specifications, including JSON Web Signature (JWS) and JSON Web Encryption (JWE), to serve this purpose. replace_me uses JWS to sign message and JWE to encrypt the signed message. JOSE actually serves as a locked briefcase containing the data (the API message) within the HTTPS “tunnel”. In other word, the communication has double protection.

X.509 Version 3 Public Key Certificate

In cryptography, X.509 is a standard that defines the format of public key certificates. JOSE makes use of this X.509 Version 3 standard for its encryption and decryption mechanism. (Indeed, X.509 certificates are used in many Internet protocols, including SSL (Secure Sockets Layer) or TLS (Transport Layer Security) we mentioned in previous session of connection security)

  • Both replace_me and Merchant need to generate and exchange Public Key Certificate for Message Encryption purpose. The production public certificate must be generated from an authorized Certificate Authority (CA) for IT security reason.
  • The Public Key Exchange Procedure is provided by your replace_me contact point.
  • The X.509 Certificate Signature should be in 2048-bit RSA and Hashing should be in SHA-256.
  • Merchant is required to use replace_me's Public Key to encrypt a message every time it sends to replace_me and verify a signed message every time it received from replace_me. Vice versa, replace_me will use Merchant's Public Key to do the same upon message sent and received from Merchant. Details will be covered in Message Encryption section.
  • For security purposes, Merchant is required to renew Public Certificate every two years (for details, please contact your replace_me contact point).

Importance of keyID for JWS & JWE

  • According to JOSE framework, replace_me uses JSON Web Signature (JWS) standard to sign a message together with JSON Web Encryption (JWE) standard to encrypt the signed message. keyID is used when creating a JWS or JWE Object, the value of the KeyID will be obtained during Public Key Exchange between replace_me & Merchant (for details, please refer to your replace_me contact point.).
  • KeyID is also used to pair the corresponding Private/Public Key that stored at Merchant and replace_me's system respectively. Hence, Merchant may need to change the KeyID during Key Renewal (for details, please refer to refer to your replace_me contact point).

How to Sign and Encrypt Message before Submitting Request

To help you understand how to construct a Signed and Encrypted Message, let's take the Java program below as an example. Do not worry if you are not familiar with Java, the idea is to let you know the steps and all needed components:

private JWSObject signMessage(String messagePayload, KeyStore ks, String keyAlias, String keyPw)
              throws UnrecoverableKeyException, KeyStoreException, NoSuchAlgorithmException, JOSEException {
            #1  Payload payload = new Payload(messagePayload);
            
            #2  JWSHeader header = new JWSHeader.Builder(JWSAlgorithm.RS256).keyID("0001").build();
            #3  JWSObject jwsObject = new JWSObject(header, payload);
            
            #4  PrivateKey privateKey = (PrivateKey) ks.getKey(keyAlias, keyPw.toCharArray());
                JWSSigner signer = new RSASSASigner(privateKey);
            #5  jwsObject.sign(signer);
            
                return jwsObject;
            }
  1. Prepare your Message Payload, that is, the plain json request message
  2. Create JWS Header using RS256 signing algorithm and JWS keyID, in this case, 0001
  3. Create JWS Object by combining JWS Header and Message Payload
  4. Retrieve your Private Key as the signer
  5. Create Signed JWS Object by signing it with the Private Key

Next, you are going to Encrypt the Signed JWS Object:

public JWEObject getEncryptedJWEObject(JWSObject jwsObject, RSAPublicKey key)
              throws JOSEException {
            #1  final Payload jwepayload = new Payload(jwsObject.serialize());
            
            #2  final JWEHeader jweheader =
                  new JWEHeader.Builder(JWEAlgorithm.RSA_OAEP_256, EncryptionMethod.A128GCM).keyID("0001").build();
            
            #3  final JWEObject jweObject = new JWEObject(jweheader, jwepayload);
            
            #4  final JWEEncrypter encrypter = new RSAEncrypter(key);
            #5  jweObject.encrypt(encrypter);
            
                return jweObject;
            }
  1. Prepare your JWE Payload, that is, the Signed JWS Object
  2. Create JWE Header. The algorithm used to encrypt the message body is A128GCM while the algorithm used to encrypt the encryption key is RSA_OAEP_256. JWE keyID is 0001 again.
  3. Create JWE Object by combining JWE Header and JWE Payload
  4. Retrieve replace_me's Public Key as the encrypter
  5. Create Encrypted JWE Object by encrypted it with replace_me's Public Key

Yes, you are now ready to put the Encrypted JWE Object as the message body (you may need to first serialize it into String format, depends on your program code design) of any API call.

How to Decrypt Message and Verify Signature after Receiving Response

Let's look into the following example to see how you decrypt a response message from replace_me:

private String decryptMessage(String respMsgPayload, KeyStoreFactory keyStore)
              throws KeyStoreException, NoSuchAlgorithmException, CertificateException, IOException, 
              java.text.ParseException, UnrecoverableKeyException, JOSEException {
            #1  JWEObject jweObject = JWEObject.parse(respMsgPayload);
            
            #2  PrivateKey privateKey = (PrivateKey) keyStore.getPrivateKey("merchant_private_key_alias");
            
                JWEDecrypter decrypter = new RSADecrypter(privateKey);
            #3  jweObject.decrypt(decrypter);
            
            #4  String signedMessage = jweObject.getPayload().toString();
                return signedMessage;
            }
  1. Create Encrypted JWE Object by parsing the encrypted response message payload
  2. Retrieve Private Key as the decrypter
  3. Decrypt the JWE Object using your Private Key
  4. Get the Signed Message from the decrypted JWE Object

You are now able to extract the plain json message. Yet, before that, you must verify the signature to guarantee data integrity.

private String verifySignature(String signedMessage, KeyStore ks, String keyAlias)
              throws KeyStoreException, JOSEException, ParseException {
            #1  JWSObject jwsObject = JWSObject.parse(signedMessage);
            
                Certificate certificate = ks.getCertificate(keyAlias);
            #2  JWSVerifier verifier = new RSASSAVerifier((RSAPublicKey) certificate.getPublicKey());
            
            #3  if (!jwsObject.verify(verifier)) {
                  throw new ValidationException("Invalid Signature");
                }
            #4  return jwsObject.getPayload().toString();
            }
  1. Create JWS Object by parsing the Signed Message
  2. Retrieve replace_me’s Public Key as the verifier
  3. Verify the signed JWS Object. Invoke error handling if invalid signature found (depends on your code design)
  4. Get the plain json message for further actions

Data Type Overview

Data Type Control:

Data Type Allowed Characters Definition & Important Notice
String (For general field) AlphaNumeric and Symbols General field means field which is NOT a critical field. replace_me system will execute characters checking upon all string fields we received in order to tackle security vulnerability, such as Cross-site Scripting. Yet, we recommend you to try use AlphaNumeric only for most cases.
String (For critical field) 0-9 a-z A-Z - _ . Critical field is used to be either a key or search criteria in replace_me backend system and hence tight restriction is applied to the allowed characters. Moreover, the starting and ending space of the string value will be trimmed before stored in replace_me system. For example, string " example 12 34 " will be trimmed to "example 12 34". List of Critical Fields: txnRef merId posMachineId employeeId
Integer 0-9 Instead of having Max Length check for String, integer range will be checked, e.g. 0 = x = 9999

Field Mandatory Control:

Field Mandatory Type Definition & Important Notice
Mandatory Annotated with Required in field definition section. Field & value must be present in the request with valid JSON format.
Optional Without a Required tag in field definition. If you don't want to pass fields that are optional, your handler should not pass neither empty strings {"example":""} nor blank value {"example":" "}.
Conditional Required under a specific condition whose logic is always provided in the field definition if it is a Conditional Field.

Time Zone Control:

Aspect Format Definition & Important Notice
In Request Message yyyy-MM-dd'T'HH:mm:ssZ Time zone is expected to be GMT+8 (Hong Kong local time). Merchant is required to perform any necessary time zone conversion before submit request if needed.
In Response Message yyyy-MM-dd'T'HH:mm:ss±hh:mm Timezone returned in api_gw object is generated from replace_me API Gateway which located in Cloud and hence is calculated in GMT+0. On the other hand, time field in response object will be returned together with timezone information. For more details, please read each field definition carefully.

Payment QR Code Creation API

POST /payment/qrCode

This API creates QR Code token with the aim of making payment. Once this API request is submitted from merchants, replace_me payment platform will return QR Code token to Merchant according to the Common QR code specification. Merchant needs to convert the QR Code token to QR Code Image and display on its online store. replace_me will be informed systematically upon Buyer confirms payment after scanning QR code. In case the online store is a Mobile App, Buyer will be directed to its FPS Mobile Payment/Banking App for payment and there will be no QR code scanning as the QR Code token will be passed to the Buyer’s App. Likewise, replace_me will be informed systematically upon Buyer confirms payment.
